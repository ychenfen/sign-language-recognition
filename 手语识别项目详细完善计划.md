## 1. 目标与执行边界

### 1.1 Objective
在不推翻现有技术栈的前提下，把项目完善为「可稳定运行、可演示、可迭代」的手语学习系统，并确保文档、接口、前后端实现一致。

### 1.2 Scope In
- 基于现有 `Vue 3 + Element Plus + MediaPipe + Flask` 架构完善。
- 对齐前后端 API 协议与运行方式。
- 补齐配置管理、错误处理、质量验证和发布流程。
- 形成可执行的阶段计划、任务清单和验收标准。

### 1.3 Scope Out
- 不迁移到 Java/Spring Boot/MySQL 重构路线。
- 不在本轮引入大型微服务拆分。
- 不做与项目目标无关的 UI 全量重设计。

### 1.4 Done Criteria
- 文档中每个阶段都有：交付物、涉及文件、验收标准、验证命令。
- 技术栈和目录映射与当前仓库代码一致。
- 至少覆盖：运行链路、API 一致性、数据持久化、模型扩展、发布与回归。

## 2. 当前代码基线（事实盘点）

## 2.1 前端现状
- 入口：`src/main.js` + `src/App.vue`
- 主要功能组件：
  - `src/components/SignDictionary.vue`（词典/搜索/收藏/详情）
  - `src/components/SignLearning.vue`（课程学习/播放控制/学习统计）
  - `src/components/SignTest.vue`（分级测试/错题本/成就）
  - `src/components/OptimizedGestureRecognition.vue`（实时手势识别）
  - `src/components/AIChat.vue`（AI 问答助手）
  - `src/components/Community.vue`（学习社区）
  - `src/components/UserCenter.vue`（登录与学习数据中心）
  - `src/components/AdminPanel.vue`（后台管理）
- API 客户端：`src/api/gestureApi.js`（默认 `http://localhost:5001/api`）

## 2.2 后端现状
- 存在两个可运行 Flask 服务文件，功能有重叠但不一致：
  - `backend/gesture_api.py`：静态手势 + AI 建议/反馈 + 视频流
  - `backend/sign_recognition_api.py`：静态手势 + LSTM 动作识别 + 部分 AI
- 风险：前端接口契约容易命中“某个脚本支持、另一个不支持”的问题。

## 2.3 数据存储现状
- 当前主要使用 `localStorage`，且 key 命名存在并行方案：
  - 通用键：`favorites`、`learningHistory`、`wrongBook`、`users`、`currentUser`
  - 业务键：`signFavorites`、`videoLearningProgress`、`community_posts`、`communityPosts`
  - 用户隔离键：`${username}_favorites` 等
- 风险：多套命名导致数据口径不一致、统计结果漂移。

## 2.4 已识别关键问题（优先级）
- `P0` 文档架构与代码不一致（旧版 Java/Spring 方案已偏离实际）。
- `P0` 后端入口重复，API 兼容边界不清晰。
- `P0` `backend/ai_integration.py` 中存在硬编码 API Key（需立即改为环境变量）。
- `P1` API 响应结构不统一（例如 `/api/gestures` 不同脚本结构不同）。
- `P1` 前端多组件本地存储键不统一。
- `P2` 缺少自动化 smoke 测试与明确发布流程。

## 3. 目标架构（本轮推荐）

```text
Browser (Vue 3 + Element Plus)
  -> src/api/gestureApi.js
      -> Flask Unified API (:5001)
          -> MediaPipe (静态手势)
          -> TensorFlow LSTM (动态手语，可选)
          -> DashScope/Qwen (AI问答与学习建议，可选)
  -> localStorage (前端学习态)
  -> sign-dataset / backend/models (训练与模型文件)
```

### 架构决策
- 保留现有单体前端 + 单体 Flask，先做稳定化，不做重架构。
- 以 `backend/sign_recognition_api.py` 为主线整合（保留 LSTM 能力），把 `gesture_api.py` 的 AI 建议/反馈与视频流并入或做兼容代理。
- 新增统一配置层（`.env` + `config`），消除硬编码密钥与 URL。

## 6. 风险与规避

### 风险 1：后端双脚本长期并存引发回归
- 规避：主入口唯一化，另一个文件仅保留转发或废弃提示。

### 风险 2：AI 服务依赖外部网络波动
- 规避：保持离线兜底回复；健康检查返回 `ai_assistant` 状态。

### 风险 3：localStorage 历史数据迁移失败
- 规避：迁移逻辑先兼容读取旧 key，再一次性写入新 key。

### 风险 4：模型准确率与演示预期不一致
- 规避：在 UI 中明确“支持手势范围 + 置信度提示 + 建议动作”。
